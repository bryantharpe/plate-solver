## Objective

Close the solve-stage latency gap between `ps-solve` (consumed via `ps-grpc`) and
cedar-solve, currently **0.27×** (Rust ~3.7× slower; e.g. 11.9 ms vs 1.4 ms on
`2019-07-29T204726_Alt40_Azi-135_Try1.jpg`). The dominant suspected cause: the Rust port
iterates 4-star combinations in **lexicographic** order while cedar uses a deliberately
tuned **breadth-first ("brightest first")** order, so the port examines many more
combinations — each paying a pattern-key computation + DB lookup — before finding the
same match.

## Hard constraints (repeat of the handoff mandate — non-negotiable)

1. **No accuracy regression.** Never loosen a tolerance, `#[ignore]` a test, or stub a
   check. Tolerances are fixed in `openspec/IMPLEMENTATION-STATUS.md` (RA/Dec 10 arcsec,
   matched catalog IDs exact, centroids ±0.1 px).
2. **Do not touch the detect stage** (`ps-detect`). It already beats both references.
3. Latency only — not a memory audit (though Phase 1 gets the C1 memory win for free),
   not a mobile port, not an accuracy improvement.
4. **Measure before and after every phase** with the eval harness (commands below). Let
   the measurements, not assumptions, decide whether the next phase runs.

## Key technical finding (verified against the reference, simplifies Phase 1)

Cedar's `breadth_first_combinations(sequence, r)`
(`reference-solutions/cedar-solve/tetra3/breadth_first_combinations.py`) recursively
yields, for each successive rightmost element, all (r−1)-combinations of the elements
before it. For fixed r this is exactly **colexicographic (colex) order**: combinations
sorted ascending by largest index, then by second-largest, and so on. Hand-trace for
r=4, n=6 (both the Python recursion and the colex successor rule below produce this):

```
[0,1,2,3],
[0,1,2,4], [0,1,3,4], [0,2,3,4], [1,2,3,4],
[0,1,2,5], [0,1,3,5], [0,2,3,5], [1,2,3,5], [0,1,4,5],
[0,2,4,5], [1,2,4,5], [0,3,4,5], [1,3,4,5], [2,3,4,5]
```

Colex order has a trivial O(1)-state, allocation-free successor, which means fix #1
(ordering) and fix #2 (C1 lazy iteration, no 618 MiB worst-case `Vec`) are **one change**.
The following implementation was **compiled and verified byte-identical to the Python
reference for n ∈ {0, 3, 4, 5, 6, 7, 8, 10, 12, 20}** (see "Verification status" at the
top of this document); it is the intended drop-in for Task 1.1:

```rust
/// Lazily yields all 4-element combinations of `[0, n)` in the same
/// breadth-first ("brightest first") order as cedar-solve's
/// `breadth_first_combinations` (tetra3/breadth_first_combinations.py),
/// which for fixed r is colexicographic order: combinations sorted
/// ascending by largest element, then by second-largest, and so on.
/// A valid match among the brightest stars is therefore reached after
/// ~C(d,4) combos (d = its largest index) instead of potentially most
/// of C(n,4) under the old lexicographic order.
///
/// Also replaces an eager `Vec<[usize; 4]>` that allocated C(n,4)·32 B
/// up front (~618 MiB at n = 150, the bundled DB's
/// verification_stars_per_fov), so timeout/cancel checks fire at the
/// same per-combo cadence with no allocation.
struct BreadthFirstCombinations4 {
    n: usize,
    cur: [usize; 4],
    done: bool,
}

impl BreadthFirstCombinations4 {
    fn new(n: usize) -> Self {
        // done = n < 4 short-circuits the empty case AND guards the
        // bound arithmetic in next() from ever running with n < 4.
        Self { n, cur: [0, 1, 2, 3], done: n < 4 }
    }
}

impl Iterator for BreadthFirstCombinations4 {
    type Item = [usize; 4];

    fn next(&mut self) -> Option<Self::Item> {
        if self.done {
            return None;
        }
        let out = self.cur;
        // Colex successor: find the smallest position that can advance
        // (bounded by the element above it, or n at the top), bump it,
        // and reset everything below to the minimal values 0..i.
        for i in 0..4 {
            let bound = if i < 3 { self.cur[i + 1] } else { self.n };
            if self.cur[i] + 1 < bound {
                self.cur[i] += 1;
                for j in 0..i {
                    self.cur[j] = j;
                }
                return Some(out);
            }
        }
        self.done = true;
        Some(out) // the final combination is still yielded
    }
}

/// Lazy breadth-first (colex) 4-combinations of `[0, n)`. No allocation.
fn breadth_first_combinations_4(n: usize) -> BreadthFirstCombinations4 {
    BreadthFirstCombinations4::new(n)
}
```

Even though this exact code has been verified out-of-repo, the equivalence guarantee must
live in CI: Task 1.3 adds the in-repo unit test with reference-generated expected
sequences (literals provided below — no Python run needed at implementation time).

### Why index-space iteration stays correct

Python applies `breadth_first_combinations` directly to the *values* of
`pattern_centroids_inds`; the Rust port iterates combinations of *positions* `[0..n)` and
then maps through `pattern_centroids_inds` (`ps-solve/src/lib.rs:211`,
`combo.map(|c| pattern_centroids_inds[c])`). Because the Python generator selects elements
purely by sequence position, `BF(seq, 4)` equals `BF(0..len, 4)` mapped element-wise
through `seq` — the existing Rust mapping is order-preserving and needs **no change**.
Brightness order is supplied upstream: `star_centroids` arrive brightest-first, and both
cluster-busting and the vsfov filter preserve that order (see comment at
`ps-solve/src/lib.rs:148`).

  exactly **three construction sites** to update (re-grep `Solution {` to confirm none

     real-failure call sites (`return Solution::failure(SolveStatus::TooFew, …)` at ~:140

     after construction, or add a parameter — implementer's choice, but the fall-through

## Phase 1 — Lazy breadth-first (colex) combination iterator

**Goal:** replace `combinations_4` (`ps-solve/src/lib.rs:595`, eager lexicographic
`Vec<[usize;4]>`) with a lazy iterator yielding cedar's breadth-first order.

### Task 1.1 — Implement the iterator

- Replace `fn combinations_4(n) -> Vec<[usize;4]>` (`ps-solve/src/lib.rs:595–607`) with
  the **verified code block from the "Key technical finding" section above**, verbatim
  (struct + `Iterator` impl + constructor fn). Update the call site at
  `ps-solve/src/lib.rs:190` to the new name `breadth_first_combinations_4` (grep-able
  intent; keeping the old name is acceptable if its doc comment is updated to say
  breadth-first).
- `n < 4` → yields nothing (the `done: n < 4` initializer both handles the empty case
  and guards the bound arithmetic).
- The call site needs no structural change: `for combo in …` drives any `Iterator`, and
  `'outer` / `break 'outer` timeout/cancel semantics are per-combo, identical to before.
- ⚠️ `notes/C1-lazy-combinations-fix.md` contains a ready lazy-iterator scaffold but in
  **lexicographic** order — reuse its structure/correctness notes, **not its successor
  logic**. Its test-edit and verification-gate sections still apply.

### Task 1.2 — Update order-dependent unit tests

- `combinations_4_basic` (~:849): n=0, 3, 4 cases — unchanged semantics; adapt to
  iterator (`.collect::<Vec<_>>()` / `.count()`). Collecting in tests also guards against
  reintroducing an eager `Vec` return type.
- `combinations_4_five_elements` (~:858): **n=5 is a trap** — lex and colex coincide when
  n = r+1, so this test passes unchanged and proves nothing about order. Keep it, but it
  must not be the only order assertion.

### Task 1.3 — Cross-check test against the Python reference (the real order proof)

- Add a unit test asserting the full iterator output for **n=6 and n=7** (r=4) equals the
  sequence produced by the actual reference generator. The expected literals below were
  generated on 2026-07-04 by running the reference itself:
  ```bash
  python3 -c "
  import sys; sys.path.insert(0, 'reference-solutions/cedar-solve/tetra3')
  from breadth_first_combinations import breadth_first_combinations
  print(list(breadth_first_combinations(range(7), 4)))"
  ```
  Embed them in the test with a comment stating that provenance (re-run the command if
  in doubt):

  **n=6 (15 combos):**
  ```text
  [0,1,2,3], [0,1,2,4], [0,1,3,4], [0,2,3,4], [1,2,3,4],
  [0,1,2,5], [0,1,3,5], [0,2,3,5], [1,2,3,5], [0,1,4,5],
  [0,2,4,5], [1,2,4,5], [0,3,4,5], [1,3,4,5], [2,3,4,5]
  ```
  **n=7 (35 combos — the 15 above, then):**
  ```text
  [0,1,2,6], [0,1,3,6], [0,2,3,6], [1,2,3,6], [0,1,4,6],
  [0,2,4,6], [1,2,4,6], [0,3,4,6], [1,3,4,6], [2,3,4,6],
  [0,1,5,6], [0,2,5,6], [1,2,5,6], [0,3,5,6], [1,3,5,6],
  [2,3,5,6], [0,4,5,6], [1,4,5,6], [2,4,5,6], [3,4,5,6]
  ```
  (Note the structure — n=7's sequence is exactly n=6's sequence followed by all combos
  containing 6. That prefix property is the defining feature of breadth-first order and
  a good sanity assertion in its own right: `take(15)` of n=7 == all of n=6.)
- Also assert, for a couple of larger n (e.g. 10, 20): yielded count == C(n,4), and the
  *set* of combos equals the lexicographic set (order differs, membership identical).

## Phase 3 — CONDITIONAL: per-combo allocation churn

**Only if Phase 2 still shows a gap.** Impact scales with how many combos reach
verification, which Phase 1 already reduced — measure first.

### Task 3.1 — Profile before touching anything

- Profile a release solve (e.g. `perf`/flamegraph or `samply`) over the benchmark images.
  Identify whether residual time is in the verification path's allocations, the DB
  lookup, or `nearby_stars`.

### Task 3.2 — Trim allocations where the profile points

- The verification path (`ps-solve/src/lib.rs` ~:300–540, on a candidate-key hit) does
  ~20 `.collect()` calls per candidate (nearby-star filtering, catalog-vector mapping,
  matched-pair extraction). Hoist reusable scratch buffers out of the loop / replace
  intermediate `Vec`s with iterator chains where the profile shows real cost.
- Do **not** touch `candidate_keys` (243 entries/combo, measured immaterial).
- Change in small, individually gated steps: every step re-runs the accuracy gates and
  the benchmark. Numerical results must be bit-for-bit unchanged — buffer reuse must not
  reorder floating-point operations; if a refactor would change summation order, don't
  do it that way.

**Acceptance:** gates green, parity table all ✓, benchmark shows the step actually
helped (revert any step that doesn't).
