## Purpose

**What:** Implement, as a single Cargo workspace, the six already-specified OpenSpec features `feat-01`…`feat-06` — `ps-core` (math), `ps-detect` (star detection), `ps-db` (pattern database read-side), `ps-dbgen` (offline DB generation), `ps-solve` (plate solver), `ps-grpc` (PlateSolver gRPC service) — the full lost-in-space pipeline (star detection → pattern-DB lookup → attitude recovery) exposed over gRPC, with **numerical parity to the Python/Rust reference** within each spec's stated tolerances.

**Why:** The OpenSpec docs in `openspec/changes/feat-0{1..6}/` are complete and `--strict`-validated; the reference is Python (tetra3, cedar-solve) plus a partial Rust detector (cedar-detect). The product needs the whole pipeline in Rust. The specs' parity scenarios are the contract. To conserve frontier tokens, **mechanical Rust coding is delegated to a local Qwen model** (`ps-coder`); **architecture, task design, and every review/judgment stay on Opus** (`ps-judge` + this orchestrator). Slower wall-clock is accepted in exchange for cheap grunt tokens. feat-07 (mobile-runtime) is explicitly OUT of this loop (user decision: stop at gRPC); the memory-mapped DB path still lands inside `ps-db`.

**Definition of done (observable):** every task checkbox below is `[x]` **or `BLOCKED`** (Phase H may legitimately block — see the Blocked Log); `cargo build --workspace` exits 0; `cargo test --workspace` exits 0 with **every feature's parity test included and green within its stated tolerance**; `ps-grpc` builds and its ExtractCentroids-interop + end-to-end `SolveFromImage` parity tests pass; and `openspec/IMPLEMENTATION-STATUS.md` exists summarizing the six features and their parity outcomes. The implementation milestone (feat-01…feat-06, Phases 0–G) is **done as of F1 (2026-06-29): 182 tests pass / 1 ignored / 24 clippy warnings / 0 errors**, and the six completed changes were archived into `openspec/specs/` on 2026-06-30 (see Decisions Log). **Phase H (hardening & spec-gap closure) is now the active work.** The loop halts when every openable Phase H box is `[x]` (BLOCKED tasks excepted) and `cargo test --workspace` is clean.

## Loop Protocol

This loop **delegates coding to a subagent**. The orchestrator (you, on Opus) never writes feature Rust directly; it designs, delegates, judges, gates, and commits. Per iteration:

1. **Re-read this file from disk** (only durable state): Purpose/DoD, all Guardrails, the task list, and the Decisions/Blocked logs.
2. **Select** the first task whose checkbox is unchecked AND whose `deps:` are all `[x]`.
3. **Orchestrate (Opus):** read the cited spec (`openspec/changes/feat-NN/specs/*/spec.md` + `design.md`) and the cited reference doc (`reference-solutions/docs/NN-*.md`) with targeted Read/Grep — or one `Explore` subagent for heavy reading. Make every architectural decision yourself (or via `ps-judge` Job B). Write a precise **implementation brief**: exact crate/files, function signatures, the load-bearing conventions, the fixtures to use, and the objective acceptance check.
4. **Delegate coding → `ps-coder`** (Qwen): hand it the brief. It writes/edits Rust, runs the cargo gate, and returns a structured `RESULT/FILES/GATE/OUTPUT/NOTES` report. (Trivial one-line edits MAY be done inline; default to delegating.)
5. **Judge → `ps-judge`** (real Anthropic Sonnet, Job A): give it the changed files + the task AC. If `VERDICT: FAIL`, return the issues to `ps-coder` and re-judge (max **3** coder↔judge rounds). Still failing → BLOCK (step 7).
6. **Gate & land (Opus):** independently run the task's integrity gate (`cargo test -p <crate>`, build, parity) — never trust a pasted summary, never weaken the gate. On green, mark the box `[x]`, append a one-line Run Log entry, and commit the work **and** this plan together with the task's exact commit message (explicit in-scope paths). Do **not** push.
7. **Stop/continue:** all boxes `[x]` → write `openspec/IMPLEMENTATION-STATUS.md` and exit. Selected task can't reach its AC → mark it `BLOCKED`, append to the Blocked Log (reason + recommended fix), continue to the next unblocked task. Otherwise loop to step 1.

**Offline / deferred judging (no connectivity, e.g. a plane).** `tools/ralph-grind.sh --offline` (or AUTO when `api.anthropic.com` is unreachable) runs a fully-local path: the orchestrator **and** `ps-coder` are local Qwen, and `ps-judge` (real Anthropic Sonnet) is **skipped**. The full integrity gate still runs locally (build + tests + golden-fixture parity all work offline — the only correctness signal that matters for *landing* code). Differences vs the online protocol: (a) a task's `deps:` are satisfied by `[x]` **or** `[~]`; (b) tasks whose AC needs an architectural decision delegated "to `ps-judge` (Job B)" are **skipped** (they require connectivity); (c) on a green gate the task is marked `[~]` (not `[x]`), an entry is appended to `## Judge Queue`, and the work is committed normally with the task's exact message. The **next ONLINE run drains the Judge Queue first** (step 0 of the online prompt): `ps-judge` reviews each pending task's committed diff vs its AC, promoting `[~]`→`[x]` on PASS or opening a follow-up fix task on FAIL, before any new task starts. Net effect: offline work is review-**deferred**, never review-**skipped**.
